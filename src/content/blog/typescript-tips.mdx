## The Power of `satisfies`

One of the most underrated TypeScript features is the `satisfies` operator. It lets you validate that an expression matches a type without widening it:

```typescript
const palette = {
  red: [255, 0, 0],
  green: "#00ff00",
} satisfies Record<string, string | number[]>;
```

This way, `palette.red` is still typed as `number[]` (not `string | number[]`), while TypeScript still checks that the object conforms to the `Record` type.

## Discriminated Unions Over Optional Properties

Instead of making properties optional, consider using discriminated unions:

```typescript
// Instead of this:
interface Result {
  success: boolean;
  data?: string;
  error?: string;
}

// Do this:
type Result =
  | { success: true; data: string }
  | { success: false; error: string };
```

The discriminated union approach makes impossible states unrepresentable. When you check `result.success`, TypeScript automatically narrows the type.

## Template Literal Types

Template literal types are incredibly powerful for creating string pattern types:

```typescript
type EventName = `on${Capitalize<string>}`;
// Matches: "onClick", "onHover", "onSubmit", etc.
```

## `as const` for Literal Types

Use `as const` to preserve literal types in arrays and objects:

```typescript
const ROLES = ["admin", "editor", "viewer"] as const;
type Role = (typeof ROLES)[number];
// type Role = "admin" | "editor" | "viewer"
```

This is much more maintainable than defining the type and the array separately.

## Conclusion

TypeScript keeps getting better with every release. These patterns have significantly improved how I write and think about types. The key insight is that TypeScript's type system is powerful enough to encode business rules — use it to make invalid states impossible rather than just annotating variables.
